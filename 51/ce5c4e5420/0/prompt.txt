# Prompt Caching for Moltis

## Context

Moltis users are requesting prompt caching. Moltis already **tracks** cache tokens in `Usage` (`cache_read_tokens`, `cache_write_tokens`) and parses them from API responses, but never **requests** caching — no cache markers are sent in requests.

After researching OpenClaw's implementation and all providers Moltis supports, here's the landscape:

### Provider Cache Support Matrix

| Provider | Caching? | Client action needed? |
|---|---|---|
| **Anthropic (direct)** | Yes | Must send `cache_control` breakpoints + beta header |
| **OpenRouter (Anthropic models)** | Yes | Same `cache_control` blocks, passed through |
| **Bedrock (Anthropic models)** | Yes | Same `cache_control` blocks |
| **OpenAI** | Automatic | No — server-side, just works |
| **DeepSeek** | Automatic | No — server-side, reports hits in usage |
| **Ollama** | Automatic | No — KV cache reused on prefix match |
| **vLLM** | Automatic | No — APC enabled by default |
| **Local GGUF** (llama-cpp-2 bindings) | Engine-level | Already managed by llama-cpp-2 KV cache reuse |
| **Groq / Together / Gemini** | No/internal | No API control |

**Key finding:** The only provider requiring client opt-in is **Anthropic** (direct, OpenRouter, and Bedrock). Everything else either caches automatically or doesn't support it. The local GGUF backend uses Rust `llama-cpp-2` bindings (not HTTP to llama-server), so `cache_prompt` doesn't apply — KV cache reuse is handled internally by the library.

## How Anthropic Prompt Caching Works

Anthropic requires `cache_control` markers on content blocks. The API caches everything up to and including a marked block. Optimal placement:

1. **System prompt** — largest, most stable content (tools, persona, instructions)
2. **Last user turn** — the conversation prefix before the model responds

The `system` field must be sent as an **array of content blocks** (not a plain string) to attach `cache_control`. A beta header is also required: `anthropic-beta: prompt-caching-2024-07-31`.

## Plan

### Step 1: Anthropic system prompt as content-block array

**File:** `crates/providers/src/anthropic.rs` — `to_anthropic_messages()` (line 123)

Currently returns `(Option<String>, Vec<Value>)` where the system is a plain string set via `body["system"] = Value::String(sys)`.

Change to return `(Option<Vec<Value>>, Vec<Value>)`. The system text becomes a content-block array with `cache_control` on the last block:

```json
[{ "type": "text", "text": "...", "cache_control": { "type": "ephemeral" } }]
```

Update `complete()` (line 252) and `stream_with_tools()` (line 354) to set `body["system"]` as this array.

### Step 2: Add cache_control to last user turn

In the same `to_anthropic_messages()`, after building all messages, find the **last user message** and inject `cache_control: { type: "ephemeral" }` on its final content block. For simple string content, convert to content-block array:

```json
// Before: { "role": "user", "content": "hello" }
// After:  { "role": "user", "content": [{ "type": "text", "text": "hello", "cache_control": { "type": "ephemeral" } }] }
```

For multimodal content (already an array), add `cache_control` to the last block.

### Step 3: Add the beta header

In both `complete()` and `stream_with_tools()`, add:
```rust
.header("anthropic-beta", "prompt-caching-2024-07-31")
```

### Step 4: OpenRouter Anthropic models

OpenRouter Anthropic models (model ID starts with `anthropic/`) are created via `OpenAiProvider` in `crates/providers/src/lib.rs`. These use the OpenAI-compatible message format but OpenRouter passes `cache_control` through to Anthropic.

Approach: In `OpenAiProvider`'s message serialization, detect when the base URL is OpenRouter and the model starts with `anthropic/`, then inject `cache_control` blocks on system and last-user messages.

**File:** `crates/providers/src/openai.rs` — `serialize_messages_for_request()` / `prepare_request_messages()`

### Step 5: Always-on, no config toggle

OpenClaw defaults to `cacheRetention: "short"` and most users never change it. We'll always enable caching for Anthropic — it's strictly beneficial (cache misses just behave like normal requests). A config toggle can be added later if needed.

## Files to Modify

| File | Change |
|------|--------|
| `crates/providers/src/anthropic.rs` | Steps 1-3: content-block system, cache_control on last user turn, beta header |
| `crates/providers/src/openai.rs` | Step 4: cache_control for OpenRouter Anthropic models |

## What We Do NOT Need

- **OpenAI/Gemini/DeepSeek changes**: These cache automatically server-side
- **Ollama/vLLM changes**: Automatic KV cache reuse, no client action
- **Local GGUF changes**: `llama-cpp-2` handles KV cache internally
- **llama-server `cache_prompt`**: Moltis uses Rust bindings, not HTTP to llama-server
- **New ChatMessage variants**: Cache control is provider-specific serialization, not a domain concept
- **Config schema changes**: No toggle needed for MVP
- **Context pruning / heartbeat**: Advanced OpenClaw optimizations, not needed initially
- **Cache diagnostics/tracing**: Nice-to-have later
